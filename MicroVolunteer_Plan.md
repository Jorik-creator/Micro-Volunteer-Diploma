# MicroVolunteer — План проекту

**Тема:** "Розробка системи краудсорсингу соціальної підтримки на базі веб-технологій"

**Суть проекту:** Веб-платформа для з'єднання людей, що потребують допомоги у повсякденних справах (літні люди, особи з інвалідністю, ВПО, багатодітні сім'ї), з волонтерами, готовими надати швидку локальну підтримку. Акцент на мікро-волонтерстві — короткотермінових завданнях: покупка продуктів, допомога з технікою, супровід до лікаря тощо.

---

## 1. АВТЕНТИФІКАЦІЯ ТА АВТОРИЗАЦІЯ

**Опис:** Система реєстрації та входу користувачів з розмежуванням ролей. Використовує вбудовану систему аутентифікації Django з розширеною моделлю користувача (AbstractUser). При реєстрації користувач обирає тип акаунту (волонтер / отримувач допомоги), що визначає доступний функціонал та інтерфейс.

### Обов'язково:
- Реєстрація користувача (email, пароль, ім'я, прізвище)
- Вибір типу акаунту при реєстрації: "Отримувач допомоги" / "Волонтер"
- Логін / Логаут
- Зміна паролю в особистому кабінеті

### Реалізація:
- Кастомна модель `User` (extends `AbstractUser`)
- `Django Forms` для реєстрації з валідацією
- `Login` / `Logout` через `django.contrib.auth`
- Зміна пароля через `PasswordChangeView`
- Middleware для перевірки типу акаунту
- Декоратори `@login_required`, кастомні `@volunteer_required`, `@recipient_required`

### Безпека:
- CSRF-токени на всіх формах
- Хешування паролів (PBKDF2 з сіллю — вбудовано в Django)
- Захист від брутфорсу (`django-axes`: блокування після 5 невдалих спроб)
- Валідація складності пароля (`MinimumLengthValidator`, `CommonPasswordValidator`)

## 2. ПРОФІЛЬ КОРИСТУВАЧА

**Опис:** Особистий кабінет з можливістю перегляду та редагування персональної інформації. Профіль волонтера містить додаткові поля: категорії допомоги та радіус готовності. Профіль отримувача — тип ситуації та контакт екстреної особи. Геолокація зберігається як координати (широта, довгота) для подальшого розрахунку відстаней.

### Обов'язково:
- Особистий кабінет з інформацією:
  - Ім'я, прізвище
  - Email, телефон
  - Адреса
  - Дата народження
  - Фото профілю
- Редагування профілю
- Налаштування геолокації:
  - Вказати адресу вручну
  - Зберегти координати (широта, довгота)

### Для волонтерів додатково:
- Вказати категорії допомоги, які готовий надавати (ManyToMany → Category)
- Радіус готовності (5 км, 10 км, 20 км)
- Короткий опис "Про себе"
- Статус доступності (доступний / недоступний)

### Для отримувачів допомоги додатково:
- Вказати свою ситуацію (літня людина, інвалідність, ВПО, багатодітна сім'я, інше)
- Контактна особа в екстреній ситуації (ім'я, телефон)

### Реалізація:
- Сигнал `post_save` для автоматичного створення профілю при реєстрації
- `Django ModelForm` для редагування
- Завантаження фото з ресайзом (`Pillow`)
- Geocoding адреси → координати (Nominatim API або ручне введення)
- Checkbox-список категорій для волонтерів (ManyToMany через form)

### Безпека:
- Валідація завантажених зображень (тип файлу, розмір ≤ 2MB)
- Користувач редагує тільки свій профіль (перевірка `request.user`)
- Санітизація текстових полів

## 3. СТВОРЕННЯ ЗАПИТУ ДОПОМОГИ (для отримувачів)

**Опис:** Ключовий модуль платформи. Отримувачі створюють запити з деталями: що потрібно, коли, де, наскільки терміново. Запити проходять повний життєвий цикл зі зміною статусів. Геолокація запиту використовується для відображення на карті та фільтрації за відстанню.

### Обов'язково:
- Форма створення запиту:
  - **Заголовок** (коротко)
  - **Опис** (детально що потрібно)
  - **Категорія** (вибір зі списку)
  - **Дата і час** (коли потрібна допомога)
    - Можливість вибрати "Якнайшвидше" / Конкретна дата-час
  - **Тривалість** (орієнтовно: 15 хв, 30 хв, 1 год, 2 год+)
  - **Кількість волонтерів** (1, 2, 3+)
  - **Адреса** (автозаповнення з профілю або вказати іншу)
  - **Геолокація** (координати з адреси)
  - **Фото** (опціонально)
  - **Рівень терміновості** (низька, середня, висока, критична)
- Збереження запиту в БД
- Статуси запиту (state machine):
  - `active` → шукає волонтерів
  - `in_progress` → волонтер відгукнувся та підтверджений
  - `completed` → виконано
  - `cancelled` → скасовано
  - `expired` → автоматично, якщо дата пройшла

### Реалізація:
- `CreateView`, `UpdateView`, `DetailView` (Django CBV)
- Форма з вибором категорії, дати, терміновості, тривалості
- Автозаповнення адреси з профілю користувача
- Management command для автоматичного прострочення: `python manage.py expire_requests`

### Безпека:
- Тільки отримувачі можуть створювати запити (`@recipient_required`)
- Автор може редагувати/скасовувати тільки свої запити
- Валідація дати (не можна створити запит на минулу дату)
- Обмеження кількості активних запитів (max 10) — захист від спаму

## 4. ПОШУК І ПЕРЕГЛЯД ЗАПИТІВ (для волонтерів)

**Опис:** Інтерактивна карта на базі Leaflet.js з маркерами активних запитів. Волонтери можуть фільтрувати запити за різними критеріями та бачити відстань до кожного запиту. Це ключова технічна фішка проекту, що демонструє роботу з геоданними та JavaScript.

### Обов'язково:
- Список всіх активних запитів
- Відображення на карті (Leaflet.js + OpenStreetMap):
  - Маркери з запитами
  - Колір маркера залежно від терміновості:
    - 🟢 зелений — низька
    - 🟡 жовтий — середня
    - 🟠 помаранчевий — висока
    - 🔴 червоний — критична
  - При кліку на маркер → popup з коротким описом
- Фільтри:
  - **За радіусом** від місцезнаходження волонтера (5, 10, 20, 50+ км)
  - **За категорією** (мультивибір)
  - **За датою** (сьогодні, завтра, цей тиждень)
  - **За терміновістю**
  - **За тривалістю** (короткі, середні, тривалі)
- Сортування:
  - За відстанню (найближчі спочатку)
  - За датою створення (нові спочатку)
  - За терміновістю
  - За рейтингом отримувача

### Детальна сторінка запиту:
- Вся інформація про запит
- Карта з місцезнаходженням
- Відстань від волонтера до запиту
- Профіль отримувача: ім'я, фото, рейтинг (зірочки), кількість виконаних запитів
- Кнопка "Відгукнутись на запит"

### Реалізація:
- Leaflet.js карта з OpenStreetMap тайлами (безкоштовно, без API key)
- Розрахунок відстані через Haversine formula (Python-функція в `utils.py`)
- JSON endpoint для передачі даних на карту (один простий Django view, що повертає `JsonResponse`)
- `ListView` + `FilterForm` для списку запитів
- Пагінація результатів

### Безпека:
- Тільки авторизовані волонтери бачать контактні дані (`@volunteer_required`)
- Зсув координат на ~100 м для приватності (точна адреса видима тільки після підтвердження)
- Пагінація результатів (захист від DoS)

## 5. ВІДГУК НА ЗАПИТ І КООРДИНАЦІЯ

**Опис:** Процес взаємодії між волонтером та отримувачем. Волонтер відгукується на запит, отримувач підтверджує або відхиляє. Після виконання волонтер позначає запит як завершений. Система підтримує запити, де потрібно кілька волонтерів.

### Обов'язково:
- Волонтер натискає "Відгукнутись":
  - Створюється `Response` зі статусом `pending`
  - Отримувач отримує сповіщення
  - Отримувач бачить хто відгукнувся (ім'я, фото, рейтинг)
- Якщо потрібно кілька волонтерів:
  - Можуть відгукнутись декілька
  - Отримувач вибирає кого підтвердити
- Отримувач може:
  - Підтвердити волонтера → статус Response `accepted`, статус запиту `in_progress`
  - Відхилити волонтера → статус Response `rejected`
  - Скасувати запит
- Після підтвердження:
  - Волонтер бачить контактні дані отримувача (телефон, точна адреса)
  - Отримувач бачить контактні дані волонтера
- Волонтер може:
  - Позначити запит як виконаний → статус `completed`

### Реалізація:
- Модель `Response`: зв'язок волонтер ↔ запит зі статусом (`pending` / `accepted` / `rejected`)
- Views для підтвердження/відхилення (POST-запити з перевіркою прав)
- Перевірка `volunteers_needed` vs кількість прийнятих — автоматичне закриття набору

### Безпека:
- Волонтер не може відгукнутись на свій запит
- Не можна відгукнутись двічі на один запит (`unique_together`)
- Тільки автор запиту може підтвердити/відхилити волонтера
- Тільки прийнятий волонтер може позначити запит як виконаний
- Контактні дані видимі тільки після підтвердження

## 6. СИСТЕМА РЕЙТИНГІВ І ВІДГУКІВ

**Опис:** Двостороння система оцінювання для побудови довіри на платформі. Після виконання запиту обидві сторони можуть оцінити одна одну (1–5 зірок + текстовий коментар). Рейтинг відображається на сторінці профілю та впливає на довіру.

### Обов'язково:
- Після виконання запиту:
  - Отримувач оцінює волонтера (1–5 зірок + коментар)
  - Волонтер оцінює отримувача (1–5 зірок + коментар)
- Відображення рейтингу:
  - Середня оцінка користувача
  - Кількість оцінок
  - Список відгуків на сторінці профілю

### Реалізація:
- Модель `Review` з constraint `unique_together(author, help_request)`
- Форма з'являється автоматично після зміни статусу на `completed`
- Розрахунок середнього рейтингу через Django Aggregation (`Avg`)
- Відображення на сторінці профілю: зірочки + пагінований список відгуків

### Безпека:
- Відгук можна залишити тільки після виконання запиту (перевірка статусу)
- Один відгук на один запит від одного автора (`unique_together`)
- Не можна залишити відгук самому собі
- Відгук не можна видалити або редагувати (цілісність даних)
- Базовий профанітет-фільтр для коментарів

## 7. СПОВІЩЕННЯ (in-app)

**Опис:** Внутрішня система сповіщень — дзвіночок у шапці сайту з лічильником непрочитаних повідомлень. Сповіщення створюються автоматично при ключових подіях через Django Signals.

### Обов'язково:
- Дзвіночок в хедері з кількістю непрочитаних сповіщень
- Список всіх сповіщень на окремій сторінці
- Позначити як прочитане / позначити все прочитаним

### Типи сповіщень:
- Новий волонтер відгукнувся на ваш запит
- Ваш відгук підтверджено / відхилено
- Запит позначено як виконаний
- Новий відгук про вас
- Нові запити у вашому радіусі (для волонтерів)

### Реалізація:
- Модель `Notification` з типами та зв'язком до запиту
- Автоматичне створення через `post_save` сигнали на `Response`, `Review`
- Дзвіночок у `base.html` з AJAX-оновленням кількості (Fetch API)
- `Context processor` для `unread_count` у всіх шаблонах
- Окрема сторінка `ListView` зі списком сповіщень

### Безпека:
- Користувач бачить тільки свої сповіщення (фільтрація по `request.user`)

## 8. ОСОБИСТА СТАТИСТИКА ТА DASHBOARD

**Опис:** Персональна статистика для кожного типу користувача та загальна статистика платформи на головній сторінці. Графіки на базі Chart.js демонструють аналітичні можливості системи.

### Для волонтерів:
- Кількість виконаних допомог
- Загальний час волонтерства (сума тривалостей)
- Категорії, в яких найбільше допомагав
- Графік активності (по тижнях)
- Історія всіх запитів (на які відгукувався)

### Для отримувачів допомоги:
- Кількість створених запитів
- Кількість виконаних запитів
- Історія всіх запитів (з фільтрами: активні, виконані, скасовані)

### Загальне (на головній сторінці):
- Загальна кількість користувачів
- Кількість активних волонтерів
- Кількість виконаних запитів
- Рейтинг, кількість відгуків, дата реєстрації

### Реалізація:
- Django ORM: `annotate()`, `aggregate()`, `Count`, `Avg`, `Sum`
- Chart.js для побудови графіків (bar chart, line chart, doughnut chart)
- JSON endpoint для передачі даних у Chart.js

### Безпека:
- Кожен бачить тільки свою детальну статистику
- Загальна статистика не розкриває персональні дані

## 9. АДМІНІСТРАТИВНА ПАНЕЛЬ

**Опис:** Кастомізована Django Admin для управління платформою. Використовуємо потужність вбудованої адмінки, розширюючи її під потреби проекту — без написання окремої кастомної панелі.

### Обов'язково:
- Перегляд всіх користувачів (з фільтрами за типом, статусом)
- Перегляд всіх запитів (з фільтрами за статусом, категорією, датою)
- Перегляд всіх відгуків
- Можливість заблокувати/розблокувати користувача (action)
- Можливість видалити запит/відгук
- Базова статистика платформи:
  - Загальна кількість користувачів
  - Кількість активних волонтерів
  - Кількість виконаних запитів

### Реалізація:
- Кастомізація `ModelAdmin` для кожної моделі
- `list_display`, `list_filter`, `search_fields`, `actions`
- Custom actions: заблокувати користувача, змінити статус запиту

## 10. UI/UX

**Опис:** Адаптивний веб-інтерфейс на базі Django Templates + Bootstrap 5. Акцент на простоті та зручності, без зайвих анімацій та візуальних ефектів.

### Обов'язково:
- Адаптивний дизайн (Bootstrap 5)
- Головна сторінка:
  - Опис платформи
  - Як це працює (3–4 кроки)
  - Статистика (кількість користувачів, виконаних запитів)
  - Кнопки "Знайти допомогу" / "Стати волонтером"
- Навігація:
  - Хедер з меню (Головна, Запити, Карта, Профіль)
  - Дзвіночок сповіщень
  - Футер з контактами
- Форми з валідацією:
  - Перевірка формату email
  - Обов'язкові поля
  - Інформативні повідомлення про помилки
- Loading states (спінери при завантаженні)
- Toast notifications (спливаючі повідомлення про успіх/помилку)

|------|-----|------|
| email | EmailField (unique) | Електронна пошта |

| Поле | Тип | Опис |
|------|-----|------|
| categories | ManyToMany → Category | Категорії допомоги |
| radius_km | IntegerField | Радіус готовності (5, 10, 20) |

| Поле | Тип | Опис |
|------|-----|------|

| emergency_contact_name | CharField | Ім'я контактної особи |

| Поле | Тип | Опис |
|------|-----|------|
| name | CharField | Назва категорії |

| icon | CharField | CSS-клас іконки |
| description | TextField | Опис категорії |

### HelpRequest
| Поле | Тип | Опис |
|------|-----|------|
| recipient | FK → User | Автор запиту |
| title | CharField | Заголовок |
| description | TextField | Детальний опис |
| category | FK → Category | Категорія |

## 12. БЕЗПЕКА (загальне)

**Опис:** Окремий розділ для дипломної записки. Django забезпечує базовий рівень захисту, але потрібно описати та продемонструвати розуміння кожного аспекту.

| Загроза | Захист | Як реалізовано |
|---------|--------|----------------|
| CSRF | CSRF-токени | `{% csrf_token %}` на всіх формах, Django middleware |
| XSS | Автоекранування | Django Templates автоматично екранують вивід |
| SQL-ін'єкція | ORM | Django ORM параметризує всі запити |
| Брутфорс паролів | Rate limiting | `django-axes` — блокування після 5 спроб |
| Слабкі паролі | Валідатори | `MinimumLengthValidator`, `CommonPasswordValidator` |
| Несанкціонований доступ | Permissions | `@login_required`, `@volunteer_required`, `@recipient_required` |
| Завантаження шкідливих файлів | Валідація | Перевірка типу, розміру (≤ 2MB), `Pillow` для зображень |
| Витік конфігурації | Environment | `python-decouple`, `.env` файл, `SECRET_KEY` не в коді |
| Доступ до чужих даних | Object-level permissions | Перевірка `request.user == object.author` у кожному view |

**Опис:** Контейнеризація додатку для демонстрації на захисті та простого розгортання.

### Компоненти:

- **`.env`** — конфігурація (`SECRET_KEY`, `DEBUG`, `DATABASE_URL`, `ALLOWED_HOSTS`)
- **Volumes** — збереження даних БД та медіафайлів між перезапусками
- **Collectstatic** — збір статичних файлів для Nginx

## 14. ТЕСТУВАННЯ

**Опис:** Мінімум 15–20 тестів для демонстрації у дипломній роботі.
